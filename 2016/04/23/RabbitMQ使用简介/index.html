RabbitMQ使用简介

By djskl

发表于 2016-04-23

1、简介

RabbitMQ是由 LShift提供的一个 Advanced Message Queuing Protocol (AMQP)的开源实现，由以高性能、健壮以及可伸缩性出名的 Erlang写成 【引自：rabbitMQ说明文档】

RabbitMQ是一个分布式的消息管理系统，其本身类似一个消息数据库，但同时具有消息路由、推送等功能。

2、安装
在Fedora上可以通过yum直接安装，也可以参考官方安装教程进行安装。默认的安装路径在官网上有详细说明。

3、基本用法
RabbitMQ的使用是典型的生产者/消费者模式，生产者产生的消息通过exchange/routingKey路由到特定的队列，消费者直接阻塞在监听的队列上，由RabbitMQ负责消息的分发。需要说明的一点是，此处的队列不是普通的队列，首先它的大部分数据不在内存而在硬盘上，其次它还可以像缓存一样为消息指定过期时间，也可以指定消息的数量，如果超过了这个数量，消息会自动丢弃(生产者不会阻塞)。

RabbitMQ提供了3种路由规则：direct、topic、fanout。direct方式将消息直接发送到名字为routing key的队列种，topic则有点类似通配符，先将队列与通配符绑定，然后将符合规则的消息发送对应的队列，fanout则是将同一个消息发送到所有的队列。

4、其他事项
4.1 流量控制
RabbitMQ默认的内存上限是安装内存的40%，也就是说低于这个值的时候不会进行流量控制，高于这个值才会控制流量。控制流量并不是直接阻塞生产者，而是让其发送速度变慢，从用户角度看就像带宽变窄了。
如果机器上的可用内存低于总内存的40%，而持续快速的发送消息会导致内存很快耗尽而RabbitMQ崩溃，此时需要在rabbitmq.config中对可用内存进行限制，比如将可用内存设置为10%，或限制为100M：

4.2 回调函数
在消费者一端，回调函数的执行是同步的，也就是说在回调函数执行完之前，该消费者是不会去队列中取新消息的，这个是与thrift类的rpc框架不同的一点。

4.3 correlation id
仔细想想本文给出的简单例子，会发现存在一个问题，如果多个客户端共用response队列，那么我们怎么知道当前结果是哪个字符串的倒置呢？比如A先发送了字符串s1，处理时间为5min，而1min后B发送了字符串s2，处理时间为2min，那么肯定是字符串s2的结果先返回(进入response队列)，而此时A与B都阻塞在response队列上，也就是说A与B都有可能被激活，如果A被激活了，那它所得到的结果就是错误的了，此时就需要correlation id来标识当前结果是属于谁的。
